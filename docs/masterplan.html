01 — The Product

One engine, three doorways

Every mode collapses to the same core: Hermes generates the words, ffmpeg burns them onto video. That's why three modes is one build, not three.

Two problems kill posting: not knowing what to make (the blank page), and hating the edit (CapCut to trim, one site to convert, another to caption, another to sticker). Reely solves both in one chat — it can brainstorm a viral concept and hand you the finished reel, or take your clip/link and cut one. You edit any step by just replying.

Mode 1 · Generate

"give me a viral reel on why upi won." → Hermes brainstorms 2–3 viral angles, you pick one → writes the hook + script → ElevenLabs voiceover → captions over kinetic text/b-roll → a tight 14–20s reel built for the algorithm.

Mode 2 · Edit

Send raw footage → Whisper transcribes → Hermes writes captions/meme text → ffmpeg trims, captions, formats → subtitled clip or sticker pack.

Mode 3 · Clip (YouTube)

Paste a link + "2:30–3:15" (or a topic) → yt-dlp grabs only that segment → captioned vertical short of the moment.

The wow in the demo: from "I don't know what to post" to a finished 14–20s reel in one message. Then show the range — paste a 2-hour podcast link, ask for one moment (Mode 3), and a captioned clip comes back the same 40 seconds later. Three inputs, one chat window.

Why it plays to Hermes's strengths

Hermes is strong at content generation (scripting, captions, meme copy) and tool use (driving ffmpeg/yt-dlp/Whisper). Reely uses both at once and registers as a native Hermes skill — the agent reads your message, routes to the right operation chain, and returns the file. That is textbook "Way 2" eligibility: a real product running on Hermes with end users interacting through it.

Scope discipline (why this is one 8-hour build, not five)

All three modes reuse the caption engine, the ffmpeg op library, the chat gateway, the job queue, and the watermark. Mode 1 adds one TTS call. Modes 2 & 3 add one transcription call. No Remotion — kinetic captions are pure ffmpeg. That is the entire delta.

Name: "reely" is a working title. Alternates to check for domains: cliply, yapclip, cutaway, snip.chat. Pick before launch so the watermark + share loop carry a real brand.

The NL → operation schema (the security firewall)

User text never becomes a shell string. Hermes maps intent to a fixed, validated operation list; each op is executed with argument arrays, not interpolated commands. This is the single most important design choice — it kills command injection and makes chaining reliable.

[ { op:"clip", start:"2:30", end:"3:15" },
  { op:"captions", source:"whisper", style:"karaoke" },
  { op:"format", aspect:"9:16" },
  { op:"watermark", text:"made with reely", show:!isPro } ]

Convex schema (core)

jobs { id, userId, mode, source, ops[], status, inBytes, outBytes, ms, isPro }
users { id, platform, handle, freeUsedToday, isPro, refCode }
usage { userId, day, count }   // enforces the free-tier daily cap

Why no Remotion

Remotion renders React in headless Chromium — 2–3GB per render, your stated crash risk. Every caption style you need (word-by-word karaoke, meme top/bottom text, kinetic titles) is achievable with ffmpeg drawtext and .ass subtitles at a fraction of the cost. Keep Remotion out of the buildathon entirely; revisit only if you later want templated motion graphics, and then run it as a single-concurrency isolated worker.

03 — Resource & Concurrency

So it never crashes under load

Your #1 concern, answered. A capped queue with hard per-job limits — not fire-and-forget.

Queue & concurrency

  • Global concurrency = CPU cores − 1 (e.g. 3 ffmpeg jobs at once)
  • Per-user cap = 1 active job; the rest queue ("you're #4 in line")
  • Queue depth > ~20 → reject politely, don't accumulate an OOM

Per-job hard limits (reject before processing)

  • File ≤ 50MB (Telegram bot-API limit anyway)
  • Duration ≤ 5 min, resolution ≤ 1080p (downscale 4K on ingest)
  • Wall-clock timeout 60–120s → kill the whole process group
  • ffmpeg -threads 2 so one job can't grab all cores

YouTube ingest — the key infra win

Never download the whole 2-hour video. yt-dlp --download-sections "*2:30-3:15" fetches only the requested segment. Cap segment length (≤3 min), cap source duration, and reject live streams. This keeps Mode 3 as light as Mode 2.

Disk hygiene (the silent killer)

Unique temp dir per job; delete input + output immediately after send. Disk-space guard refuses new jobs below a free-space threshold. TTL sweep for orphaned files. On timeout, kill -9 the process group or zombie ffmpeg/yt-dlp procs eat RAM.

Rough capacity on one modest box (4 vCPU / 8GB)

3 concurrent ffmpeg jobs ≈ 0.6–1.2GB RAM, each caption/trim job finishes in seconds to ~30s. At ~3 jobs/min throughput that's thousands of shorts/day queued smoothly. A viral spike doesn't crash it — it queues, and "you're #6 in line" is acceptable UX. What would crash it (Remotion, whole-video downloads, uncapped file sizes, no timeouts) is exactly what the rules above remove.

Contrast with died.fyi: there, virality directly multiplied browser-farm cost. Here, the caps + segment-only downloads + ffmpeg-first keep marginal cost per short at cents, and the queue absorbs spikes. This is the light shape you wanted.

Hosting & cost — one VPS is enough

Hermes is the Telegram/WhatsApp backend — you don't build a bot server. Its gateway connects to chat, routes to the reely skill, and sends the file back. Hermes runs on the same VPS as the ffmpeg workers. The model is an external OpenAI-compatible API call, so no GPU is needed — that's what keeps hosting cheap.

05 — Virality & Revenue

Winning the track + the ₹99 unlock

Signups are 25×. A private tool has no free loop — so we engineer three, and one feature is both the ad and the revenue.

Loop 1 · Watermark = ad + revenue

Free outputs carry "made with reely". Every reel someone posts is a distributed ad. Removing it is the paid unlock — the single most proven micro-payment pattern (Epik, upscalers all monetized exactly this).

Loop 2 · Stickers in-chat

Sticker/GIF output spreads through WhatsApp & Telegram groups by design — the one output that propagates inside the app it's built on.

Loop 3 · Build-in-public + signup gate

The "I replaced CapCut + 4 sites with one chat" post is strong builder-Twitter content. Pair it with "3 free shorts, then sign up to continue" — that turns reach into the 25× meaningful-action metric.
